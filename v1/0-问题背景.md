# 问题定义（Problem Statement）

## 一、问题背景（Background）
 当前开发环境： ChatGPT Web + Git + Codex Desktop + Solo 

目标：开发一款完整的软件产品，实现从：

```text
需求分析
↓

产品设计
↓

架构设计
↓

编码实现
↓

测试验证
↓

部署发布
↓

产品上线
↓

持续迭代
```

整个生命周期均由 AI 协助完成。

其中：

* **ChatGPT** 负责需求分析、产品设计、架构设计、任务拆解、技术方案设计等高层工作。
* **Codex** 负责代码实现、测试、调试、重构等工程执行工作。

理想状态下，希望整个研发流程形成连续、可迭代、低人工干预的协作闭环。

---

# 二、当前工作流（As-Is）

目前采用的研发流程如下：

```text
ChatGPT

↓

需求分析

↓

输出 Markdown 文档

↓

人工保存 md 文件

↓

打开 Codex

↓

复制 Prompt 或 md 内容

↓

Codex 编码

↓

人工 Review

↓

返回 ChatGPT

↓

继续分析

↓

再次复制 Prompt

↓

Codex 继续开发
```

整个开发周期需要频繁在 ChatGPT、文件系统、GitHub、Codex 之间切换。

---

# 三、存在的问题（Pain Points）

当前流程存在的问题，并不仅仅是 Markdown 文件传递，而是整个研发流程缺乏统一协作机制。

## Problem 1：ChatGPT 与 Codex 没有共享上下文

ChatGPT 已完成：

* 产品定位
* 用户需求分析
* 系统架构设计
* 数据模型设计
* API 设计
* 开发计划

但 Codex 无法直接获得这些上下文。

因此每进入一个开发任务，都需要：

* 重新复制
* 重新解释
* 重新理解

造成大量重复沟通。

---

## Problem 2：人工复制 Prompt 成为主要流程成本

目前几乎每一个开发阶段都需要：

```text
复制

↓

切换窗口

↓

粘贴

↓

重新说明需求
```

这种成本不是技术问题，而是流程问题。

随着项目规模扩大，重复操作越来越多，效率持续下降。

---

## Problem 3：缺少统一的项目状态管理

项目中同时存在多个状态来源：

* ChatGPT 记录的任务进度
* Codex 当前开发状态
* GitHub 当前代码状态
* 本地文档版本

这些状态彼此独立，没有统一同步。

最终容易出现：

* ChatGPT 认为做到 Task 8
* Codex 实际做到 Task 6
* GitHub 已 Merge Task 7
* 人工已经无法确认真实状态

项目状态逐渐失控。

## Problem 6：AI 工具之间没有真正协作

目前：

* ChatGPT
* Codex
* GitHub

实际上彼此独立。

它们只是被人工串联。

整个研发流程仍然依赖开发者不断切换工具。

AI 工具没有形成真正意义上的协同研发能力。

## Problem 7：一个人的产品研发流程是否合理？
目前：

- 依赖传统产品研发流程，不知道在AI Coding时代，最佳实践应该是什么样的？
- 需要制定适合 ChatGPT ↔ Codex ↔ Git ↔ Solo 组合的 产品研发流程，或者协议；
- 流程如何涉及多个阶段，需要规范每个阶段的输出物、输入物标准
- 需要知道 流程进度信息
- 需要知道 流程导航信息：当前是什么阶段、下一步是什么阶段
- 每个阶段的完成不应该由人来判断
- 如果流程需要状态，状态不应该由人来维护
- 如果执行环境切换，必须提供明确的提示，包括是否是新会话、新会话Bootstrap是什么，新会话的所依赖的上下文是什么？如果上下文依赖的是git仓库的文件，最好避免让人工来上传。


## Problem 8:人 + ChatGPT + Codex + Git之间缺少工程协议（Engineering Protocol）
目前：
- 需求分析、产品设计、架构设计、任务生成阶段：在ChatGPT会话中完成，形成结论，由人工复制到本地文件，手动提交至GIT
- 编码实现阶段：人工复制 TASK 内容 到 Codex中执行 并自动提交至Git
- 测试验证阶段：该阶段还没有最佳实践。

# 四、问题本质（Root Cause）

经过分析，可以发现：

真正的问题不是：

> ChatGPT 不能直接向 Codex 发送 Prompt。

真正的问题是：

> **整个研发流程缺乏统一的共享上下文（Shared Context）与统一状态管理（Single Source of Truth）。**

由于没有统一的数据源：

* ChatGPT 无法持续了解项目状态。
* Codex 无法理解完整产品背景。
* GitHub 无法承担产品知识中心。

最终导致所有 AI Agent 都只能完成局部工作，而无法持续协同。

---

# 五、真正需要解决的问题（Problem Definition）

因此，本项目真正要解决的问题不是：

> 如何减少 Markdown 文件复制粘贴。

也不是：

> 如何让 ChatGPT 直接调用 Codex。

真正的问题应该重新定义为：

> **如何构建一个面向 Solo Developer 的 AI 协同产品研发系统，使 ChatGPT、Codex、GitHub 能够共享统一上下文、共享统一状态，并形成持续、可追踪、可迭代的软件研发闭环。**

最终目标包括：

1. 建立统一的（Single Source of Truth）。
2. 消除重复复制与人工同步成本。
3. 实现 AI Agent 之间的持续协作。
4. 建立完整的软件研发闭环。
5. 支持长期维护、持续迭代与产品演进，而不是一次性代码生成。
